153 /*
154  * Renumbers RG numbers in a cram compression header.
155  *
156  * CRAM stores RG as the Nth number in the header, rather than a
157  * string holding the ID: tag. This is smaller in space, but means
158  * "samtools cat" to join files together that contain single but
159  * different RG lines needs a way of renumbering them.
160  *
161  * The file descriptor is expected to be immediately after the
162  * cram_container structure (ie before the cram compression header).
163  * Due to the nature of the CRAM format, this needs to read and write
164  * the blocks itself. Note that there may be multiple slices within
165  * the container, meaning multiple compression headers to manipulate.
166  * Changing RG may change the size of the compression header and
167  * therefore the length field in the container. Hence we rewrite all
168  * blocks just incase and also emit the adjusted container.
169  *
170  * The current implementation can only cope with renumbering a single
171  * RG (and only then if it is using HUFFMAN or BETA codecs). In
172  * theory it *may* be possible to renumber multiple RGs if they use
173  * HUFFMAN to the CORE block or use an external block unshared by any
174  * other data series. So we have an API that can be upgraded to
175  * support this, but do not implement it for now. An example
176  * implementation of RG as an EXTERNAL block would be to find that
177  * block and rewrite it, returning the number of blocks consumed.
178  *
179  * Returns 0 on success;
180  * -1 if unable to edit;
181  * -2 on other errors (eg I/O).
182  */
183 int cram_transcode_rg(cram_fd *in, cram_fd *out,
184  cram_container *c,
185  int nrg, int *in_rg, int *out_rg);
